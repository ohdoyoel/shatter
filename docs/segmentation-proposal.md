## 1. 현재 접근 방식의 문제점

### 1.1 현재 알고리즘 요약

현재 `capture.ts`의 분할 알고리즘은 다음과 같이 동작한다:

1. DOM 트리를 순회하며 각 요소의 `getBoundingClientRect()`로 바운딩 박스를 얻는다.
2. 면적 기준으로 분류한다: 뷰포트의 0.01% 미만(너무 작음, 건너뜀), 5% 초과(너무 큼, 자식으로 재귀), 그 사이(캡처 대상).
3. 배경이 투명하거나 body 배경과 동일한 요소는 자식으로 재귀하거나 건너뛴다.
4. `element-filter.ts`에서 중복 제거 및 최대 150개로 제한한다.
5. 클라이언트에서 각 요소의 바운딩 박스를 전체 스크린샷의 `background-position`으로 잘라 보여준다.

### 1.2 근본적인 문제들

#### (1) DOM 구조와 시각적 구조의 불일치

DOM은 **의미론적(semantic)** 구조이지 **시각적** 구조가 아니다. CSS `flexbox`, `grid`, `position: absolute`, `transform`, `overflow` 등으로 인해 DOM 계층 구조와 실제 화면에 보이는 시각적 배치가 전혀 다를 수 있다. 하나의 시각적 카드가 5개의 중첩된 `<div>`로 구성될 수 있고, 반대로 하나의 `<div>`가 시각적으로 완전히 분리된 여러 영역을 포함할 수도 있다.

#### (2) 바운딩 박스는 직사각형 강제

모든 요소가 직사각형 바운딩 박스로 추출된다. 실제 웹페이지의 시각적 요소는 `border-radius`로 둥글거나, `clip-path`로 잘려 있거나, 불규칙한 형태일 수 있다. 직사각형 조각으로 깨지는 것은 "유리가 깨지는" 느낌과 거리가 멀고, "포스트잇을 떼어내는" 느낌에 더 가깝다.

#### (3) 겹침(Overlap)과 빈 공간(Gap) 문제

DOM 요소의 바운딩 박스는 서로 겹치거나, 사이에 빈 공간을 남긴다:

- **겹침**: 부모와 자식, 또는 형제 요소의 바운딩 박스가 겹치면 같은 영역이 여러 조각에 포함된다. 현재 `filterElements`에서 위치/크기 기반 중복 제거(5px 이내)를 하지만, 부분적으로 겹치는 경우는 처리하지 못한다.
- **빈 공간**: 요소 사이의 `margin`, `gap`, 또는 배경만 있는 영역은 어떤 조각에도 포함되지 않는다. Shatter 후 이 영역은 그냥 사라지거나 배경이 그대로 노출되어 부자연스럽다.

#### (4) "깨진다"기보다 "분리된다"에 가까운 시각 효과

현재 방식에서 요소들은 원래 위치에서 직사각형 그대로 떨어진다. 이것은 "유리가 산산이 부서지는" 느낌이 아니라, "선반에서 책이 떨어지는" 느낌이다. 실제 "shatter" 효과를 내려면:
- 조각의 형태가 불규칙해야 한다.
- 페이지 전체가 빈틈없이 조각으로 채워져야 한다(tessellation).
- 깨지는 지점에서 방사형으로 균열이 퍼져야 한다.

#### (5) 사이트 의존적인 불안정한 결과

동일한 "카드 UI"라도 사이트마다 DOM 구조가 완전히 다르다. 어떤 사이트에서는 카드 하나가 하나의 요소로 캡처되고, 다른 사이트에서는 10개의 조각으로 쪼개진다. 면적 기반 임계값(`MIN_AREA`, `MAX_AREA`)은 사이트 레이아웃에 따라 적절할 수도, 완전히 부적절할 수도 있다.

#### (6) 성능과 정밀도의 트레이드오프

150개 요소 제한은 성능을 위해 필요하지만, 정교한 페이지에서는 중요한 요소가 잘리고, 단순한 페이지에서는 너무 큰 덩어리로 떨어진다. 면적 기반 필터링은 시각적 중요도와 무관하게 작동한다.

## 2. 새로운 접근 방식 제안

### 접근법 A: Voronoi/Delaunay 기반 불규칙 분할 (실현 가능성: ★★★★★)

#### 개요

전체 페이지 스크린샷 위에 Voronoi 다이어그램을 생성하여 페이지를 불규칙한 다각형 조각으로 분할한다. DOM 구조를 완전히 무시하고, 순수하게 기하학적으로 "깨뜨린다."

#### 작동 방식

1. **시드 포인트 생성**: 페이지 영역에 랜덤 포인트를 뿌린다 (예: 100~200개). 선택적으로 Poisson Disk Sampling을 사용하여 균일한 분포를 보장한다.
2. **Voronoi 다이어그램 계산**: `d3-delaunay` 라이브러리로 Voronoi 셀을 계산한다.
3. **균열 효과(선택)**: 클릭/탭 지점에서 밀도를 높여 방사형 균열 패턴을 만들 수 있다.
4. **스크린샷 매핑**: 각 Voronoi 셀의 다각형 클리핑 경로(`clip-path: polygon(...)`)를 사용하여 전체 스크린샷에서 해당 영역만 보이게 한다.
5. **물리 바디 생성**: 각 셀을 Matter.js의 `Bodies.fromVertices()`로 생성한다.

#### 장점

- DOM 구조에 의존하지 않으므로 **모든 사이트에서 일관된 결과**를 낸다.
- 불규칙한 다각형 조각이 **진짜 깨지는 느낌**을 준다.
- 페이지 전체를 빈틈없이 커버하므로 **빈 공간 문제가 없다**.
- 구현이 비교적 단순하다 (Voronoi 계산은 수 밀리초).
- 시드 포인트 수로 조각 개수를 정밀하게 제어할 수 있다.

#### 단점

- 텍스트, 이미지 등이 의미 없이 잘린다 (글자 중간에서 잘릴 수 있음).
- Matter.js의 `Bodies.fromVertices()`는 오목(concave) 다각형을 자동으로 볼록(convex) 분해하므로 약간의 성능 비용이 있다.
- 시각적 콘텐츠의 맥락을 완전히 무시한다.

### 접근법 B: 콘텐츠 인식 Voronoi (시각 에지 가이드) (실현 가능성: ★★★★☆)

#### 개요

접근법 A의 Voronoi를 기반으로 하되, 시드 포인트 배치를 스크린샷의 시각적 에지(edge)에 따라 가이드한다. 조각의 경계가 자연스럽게 콘텐츠의 경계를 따르도록 한다.

#### 작동 방식

1. **에지 검출**: 서버 사이드에서 `sharp` 라이브러리로 스크린샷을 그레이스케일 변환 후, Sobel 또는 Canny 에지 검출을 수행한다 (또는 간단히 인접 픽셀 차이 기반 에지 맵).
2. **시드 포인트 분포 조정**: 에지 밀도가 높은 영역에는 시드 포인트를 밀집 배치하고, 균일한 배경 영역에는 듬성듬성 배치한다. 이렇게 하면 텍스트/이미지 경계에서는 작은 조각, 빈 배경에서는 큰 조각이 생긴다.
3. **Voronoi 계산 및 렌더링**: 접근법 A와 동일.

#### 장점

- 접근법 A의 모든 장점을 유지하면서, **시각적으로 더 자연스러운 분할**.
- 콘텐츠가 있는 영역은 작은 조각으로 디테일을 살리고, 빈 영역은 큰 조각으로 성능을 아낀다.
- 에지 검출은 `sharp`로 매우 빠르게 수행 가능 (수십 ms 수준).

#### 단점

- 에지 검출 정확도에 따라 결과 품질이 달라진다.
- 구현 복잡도가 약간 증가한다.
- 여전히 텍스트 중간에서 잘릴 수 있다 (다만 빈도가 줄어든다).

### 접근법 C: 하이브리드 DOM + Voronoi (실현 가능성: ★★★☆☆)

#### 개요

DOM 요소의 바운딩 박스를 "힌트"로 사용하되, 최종 분할은 Voronoi로 수행한다. 주요 콘텐츠 블록(이미지, 버튼 등)은 통째로 보존하면서, 나머지 영역은 Voronoi로 불규칙하게 분할한다.

#### 작동 방식

1. **DOM 분석**: 현재 알고리즘과 유사하게 주요 시각 요소(이미지, 버튼, 아이콘 등)를 식별한다.
2. **보존 영역 지정**: 식별된 요소의 바운딩 박스를 "보존 영역"으로 마킹한다.
3. **나머지 영역 Voronoi 분할**: 보존 영역을 제외한 나머지 공간에 Voronoi 시드를 뿌려 분할한다.
4. **병합**: 보존된 직사각형 요소 + Voronoi 다각형 조각을 합쳐 최종 분할 결과를 만든다.

#### 장점

- 의미 있는 콘텐츠(로고, 버튼, 이미지)가 잘리지 않는다.
- 나머지 영역은 불규칙한 파편으로 "깨지는" 느낌을 준다.
- 두 가지 시각 효과를 조합할 수 있다.

#### 단점

- **구현 복잡도가 높다**: Voronoi 셀이 보존 영역을 침범하지 않도록 클리핑해야 한다.
- 보존 영역과 Voronoi 셀 사이의 경계 처리가 까다롭다.
- DOM 의존성이 남아있어 사이트에 따라 결과가 달라질 수 있다.
- 직사각형 조각과 불규칙 조각이 섞여 시각적 일관성이 떨어질 수 있다.

### 접근법 D: 순수 시각 분할 (Connected Components) (실현 가능성: ★★☆☆☆)

#### 개요

스크린샷을 이미지 처리하여 시각적으로 구분되는 영역(connected components)을 자동 검출한다.

#### 작동 방식

1. **에지 검출**: Canny 또는 Sobel 필터로 에지 맵을 생성한다.
2. **영역 분할**: 에지로 둘러싸인 연결된 영역(connected components)을 식별한다.
3. **바운딩 영역 추출**: 각 connected component의 외곽선을 다각형으로 근사한다.
4. **물리 바디 생성**: 각 다각형을 Matter.js 바디로 변환한다.

#### 장점

- DOM에 전혀 의존하지 않는다.
- 시각적으로 의미 있는 단위로 분할된다.

#### 단점

- **브라우저에서 실시간 이미지 처리가 무겁다**: OpenCV.js는 ~8MB이며 로딩만 수 초.
- 에지 검출 파라미터 튜닝이 사이트마다 달라야 한다.
- 텍스트가 글자 단위로 분리되거나, 배경과 비슷한 색의 요소가 병합될 수 있다.
- 빈 배경 영역은 하나의 거대한 조각이 되어 의미 없다.
- 2~3초 성능 목표 내에 처리가 어렵다.

### 접근법 E: 그리드 + 노이즈 기반 타일링 (실현 가능성: ★★★★☆)

#### 개요

정규 그리드를 기반으로 하되, 각 격자점에 Perlin/Simplex 노이즈로 변위를 주어 불규칙한 타일 분할을 만든다. Voronoi보다 계산이 단순하면서도 비슷한 시각 효과를 낸다.

#### 작동 방식

1. **기본 그리드 생성**: 페이지를 N x M 격자로 나눈다 (예: 10x15).
2. **노이즈 변위 적용**: 각 격자 교차점에 Simplex 노이즈 기반의 랜덤 변위를 적용한다.
3. **삼각형/사각형 분할**: 변위된 격자점들을 연결하여 불규칙한 사각형 또는 삼각형 타일을 만든다.
4. **렌더링**: 각 타일을 `clip-path`로 잘라 스크린샷 조각으로 표시한다.

#### 장점

- **매우 빠르다**: 순수 수학 연산, 라이브러리 불필요.
- **조각 수를 정확히 제어**할 수 있다 (격자 크기로).
- 삼각형 타일은 물리 엔진에서 자연스럽게 볼록 다각형이므로 `fromVertices` 분해가 필요 없다.
- 구현이 단순하다.

#### 단점

- 시각적 콘텐츠를 전혀 고려하지 않는다.
- 격자 기반이라 완전한 Voronoi만큼 "깨지는" 느낌이 자연스럽지 않다.

### 접근법 F: 방사형 균열 패턴 (Radial Fracture) (실현 가능성: ★★★★☆)

#### 개요

실제 유리가 깨지는 패턴을 시뮬레이션한다. 충격 지점에서 방사형 균열과 동심원 균열이 교차하며 불규칙한 조각을 만든다.

#### 작동 방식

1. **충격 지점 결정**: 사용자의 클릭/탭 위치 또는 화면 중앙.
2. **방사형 선 생성**: 충격 지점에서 랜덤 각도로 20~30개의 방사선을 발사한다. 각 선에 약간의 곡률(Perlin 노이즈)을 추가한다.
3. **동심원 링 생성**: 충격 지점을 중심으로 불규칙한 간격의 동심원을 그린다.
4. **교차 다각형 계산**: 방사선과 동심원이 교차하여 만드는 다각형들을 계산한다.
5. **렌더링**: 각 다각형을 `clip-path`로 적용.

#### 장점

- **가장 실감나는 "shatter" 효과**: 실제 유리/도자기 파쇄 패턴과 유사.
- 충격 지점 근처는 작은 조각, 멀어질수록 큰 조각 (자연스러운 밀도 분포).
- DOM이나 이미지 분석에 의존하지 않아 **일관적이고 빠르다**.
- 사용자 인터랙션(클릭 위치)과 자연스럽게 연결된다.

#### 단점

- 다각형 계산 로직이 다소 복잡하다 (선 교차, 영역 계산).
- 페이지 가장자리에서 다각형이 부자연스럽게 잘릴 수 있다.
- 콘텐츠를 전혀 고려하지 않는다.

## 3. 구현 시 고려사항

### 3.1 서버 vs 클라이언트 처리

| 작업 | 서버 (Node.js) | 클라이언트 (브라우저) | 권장 |
|------|----------------|----------------------|------|
| Voronoi/기하 계산 | 가능, 빠름 | 가능, 빠름 | **클라이언트** (전송 데이터 감소) |
| 에지 검출 (sharp) | 매우 빠름 (native) | 느림 (OpenCV.js 8MB) | **서버** |
| 스크린샷 촬영 | 서버 전용 (Puppeteer) | 불가 | **서버** |
| clip-path 생성 | 가능 | 가능 | **클라이언트** (DOM 조작) |
| Matter.js 바디 생성 | 불가 | 클라이언트 전용 | **클라이언트** |

**권장 분리**: 서버에서는 스크린샷 + (선택적) 에지 맵만 생성하고, 기하학적 분할 계산과 렌더링은 모두 클라이언트에서 수행한다. 이렇게 하면 서버 응답 크기가 최소화되고, 분할 파라미터(조각 수, 충격 위치 등)를 클라이언트에서 실시간으로 조절할 수 있다.

### 3.2 성능 (2~3초 목표)

현재 파이프라인에서 병목은 Puppeteer의 페이지 로딩(~5~15초)과 스크린샷 촬영(~1초)이다. 분할 알고리즘 자체는 전체 시간의 극히 일부만 차지하므로, 대부분의 접근법이 성능 요건을 충족한다:

| 접근법 | 분할 계산 시간 (추정) | 추가 라이브러리 크기 |
|--------|---------------------|---------------------|
| A. Voronoi | ~5ms | d3-delaunay: ~30KB |
| B. 콘텐츠 인식 Voronoi | ~50ms (에지검출 서버) + ~5ms | sharp (서버, 이미 설치 가능) + d3-delaunay |
| C. 하이브리드 | ~100ms (DOM 분석) + ~10ms | d3-delaunay + DOM 코드 유지 |
| D. Connected Components | ~500ms~2s | OpenCV.js: ~8MB |
| E. 그리드+노이즈 | ~2ms | 없음 (순수 수학) |
| F. 방사형 균열 | ~10ms | 없음 (순수 수학) |

### 3.3 Matter.js 물리 바디 관련

#### 볼록(Convex) vs 오목(Concave) 다각형

Matter.js는 기본적으로 **볼록 다각형**만 지원한다. `Bodies.fromVertices()`에 오목 다각형을 전달하면 내부적으로 `poly-decomp` 라이브러리를 사용해 볼록 분해를 수행한다.

| 조각 형태 | Matter.js 호환성 | 성능 |
|-----------|-----------------|------|
| 직사각형 | `Bodies.rectangle()` - 완벽 | 최고 |
| 볼록 다각형 | `Bodies.fromVertices()` - 완벽 | 좋음 |
| 오목 다각형 | `Bodies.fromVertices()` + `poly-decomp` | 보통 |
| 임의 다각형 (100+개) | 분해 + 복합 바디 | 무거움 |

**핵심**: Voronoi 셀은 항상 **볼록 다각형**이므로 `poly-decomp` 없이 직접 `Bodies.fromVertices()`로 생성할 수 있다. 이는 Voronoi 기반 접근법의 큰 장점이다.

삼각형 분할(접근법 E의 변형)도 항상 볼록이므로 물리 엔진과의 호환성이 완벽하다.

방사형 균열(접근법 F)의 조각은 대부분 볼록이지만, 동심원 호가 포함된 조각은 오목할 수 있다. 이 경우 호를 직선 세그먼트로 근사하면 해결된다.

#### 렌더링 동기화

현재 방식은 각 요소의 DOM `<div>`에 `transform: translate() rotate()`를 적용하여 물리 바디 위치를 반영한다. 새로운 접근법에서도 동일한 패턴을 사용할 수 있다:

```
각 조각 div:
  - position: absolute
  - left/top: 조각의 초기 위치
  - clip-path: polygon(...) 으로 형태 정의
  - background-image: 전체 스크린샷
  - background-position: 해당 조각의 오프셋
  - transform: translate(dx, dy) rotate(angle) 으로 물리 위치 반영
```

이 패턴은 현재 `PhysicsScene.tsx`의 구조와 완전히 호환된다.

### 3.4 활용 가능한 라이브러리

| 라이브러리 | 용도 | 크기 | 비고 |
|-----------|------|------|------|
| **d3-delaunay** | Voronoi 다이어그램 | ~30KB | 가장 빠르고 안정적인 Voronoi 구현 |
| **sharp** | 서버사이드 이미지 처리 (에지 검출) | native | 이미 Node.js 생태계에서 표준 |
| **poly-decomp** | 오목 다각형 → 볼록 분해 | ~10KB | Matter.js가 이미 선택적 의존성으로 지원 |
| **simplex-noise** | Perlin/Simplex 노이즈 | ~5KB | 격자 변위, 균열선 곡률에 사용 |
| **poisson-disk-sampling** | 균일 랜덤 포인트 분포 | ~5KB | Voronoi 시드 포인트 배치에 사용 |
| **OpenCV.js** | 범용 이미지 처리 | ~8MB | 너무 무거움, 권장하지 않음 |

## 4. 최종 권장안

### 권장: 접근법 F (방사형 균열) + 접근법 A (Voronoi) 조합

**"방사형 균열 Voronoi (Radial Fracture Voronoi)"**

#### 왜 이 조합인가

1. **시각적 임팩트가 가장 크다**: 사용자가 클릭한 지점에서 실제 유리가 깨지듯 균열이 퍼지는 것은 "shatter"라는 앱 이름과 가장 잘 맞는 경험이다.

2. **기술적으로 가장 깔끔하다**: DOM 분석을 완전히 제거하고, 스크린샷 하나만으로 동작한다. 서버에서 보내는 데이터가 단순해지고(스크린샷 + 메타데이터), 사이트별 분기 로직이 사라진다.

3. **성능이 우수하다**: Voronoi 계산은 수 밀리초, 추가 라이브러리는 d3-delaunay (~30KB) 하나면 충분하다. 현재 DOM 순회 + HTML 클론 + CSS 인라이닝에 소요되는 시간을 완전히 절약할 수 있다.

4. **물리 엔진과의 호환성이 완벽하다**: Voronoi 셀은 항상 볼록 다각형이므로 `poly-decomp` 없이 Matter.js에서 직접 사용할 수 있다.

5. **일관성이 보장된다**: 어떤 사이트를 넣어도 동일한 품질의 shatter 효과를 얻는다. DOM 구조에 따라 결과가 들쭉날쭉하던 문제가 완전히 해결된다.

#### 구현 전략 개요

##### 서버 (capture.ts 변경)

```
변경 전: URL → Puppeteer → DOM 순회 → 요소 추출 → HTML 클론/인라이닝 → CaptureResponse
변경 후: URL → Puppeteer → 스크린샷만 촬영 → CaptureResponse (스크린샷 + 메타데이터만)
```

서버의 역할이 대폭 단순해진다. DOM 순회, HTML 클론, CSS 인라이닝 코드를 모두 제거할 수 있다. `element-filter.ts`도 불필요해진다.

##### 클라이언트 (새로운 흐름)

```
1. 스크린샷 수신
2. 사용자 클릭 대기 (또는 자동 트리거)
3. 클릭 지점을 중심으로 Voronoi 시드 생성:
   - 중심부: 밀집 배치 (작은 조각)
   - 외곽부: 희소 배치 (큰 조각)
   - Poisson Disk Sampling으로 최소 간격 보장
4. d3-delaunay로 Voronoi 다이어그램 계산
5. 각 Voronoi 셀에 대해:
   - <div> 생성
   - clip-path: polygon(...) 적용
   - background-image: 전체 스크린샷
   - background-position: 셀 위치에 맞게 설정
   - Matter.js Bodies.fromVertices()로 물리 바디 생성
6. 물리 시뮬레이션 시작 + 초기 폭발 임펄스
```

##### 데이터 구조 변경

```typescript
// 변경 전
interface CapturedElement {
  id: string;
  x: number;
  y: number;
  width: number;
  height: number;
  isStatic: boolean;
}

// 변경 후
interface ShatterFragment {
  id: string;
  vertices: { x: number; y: number }[];  // Voronoi 셀의 꼭짓점들
  centroid: { x: number; y: number };     // 질량 중심 (물리 바디 위치)
  boundingBox: { x: number; y: number; width: number; height: number }; // clip-path용
}
```

##### 시각적 개선 포인트

- **균열 애니메이션**: shatter 전에 균열선이 퍼지는 0.3초짜리 CSS 애니메이션을 추가하면 임팩트가 극적으로 상승한다.
- **조각별 지연**: 중심부 조각부터 시간차를 두고 떨어지게 하면 충격파 효과를 줄 수 있다.
- **입자 효과**: 작은 삼각형 파편을 추가로 뿌리면 "먼지" 효과로 더 실감난다.
- **소리**: 유리 깨지는 사운드 이펙트를 추가할 수 있다.

#### 개발 단계 제안

| 단계 | 작업 | 예상 시간 |
|------|------|-----------|
| 1 | 서버 단순화 (스크린샷만 반환) | 2시간 |
| 2 | 클라이언트 Voronoi 분할 구현 (d3-delaunay) | 4시간 |
| 3 | clip-path + background 기반 조각 렌더링 | 3시간 |
| 4 | Matter.js fromVertices 바디 생성 및 물리 연결 | 3시간 |
| 5 | 방사형 시드 분포 (중심 밀집) 구현 | 2시간 |
| 6 | 균열 애니메이션 및 시각 효과 | 3시간 |
| 7 | 테스트 및 성능 최적화 | 3시간 |
| **합계** | | **~20시간** |

## 부록: 현재 코드에서 제거 가능한 부분

새로운 접근법을 채택하면 다음 코드가 불필요해진다:

- `lib/capture.ts`: DOM 순회 로직 (141~286행) 전체를 스크린샷 촬영만으로 대체
- `lib/element-filter.ts`: 파일 전체 삭제 가능
- `lib/types.ts`: `CapturedElement` 인터페이스를 `ShatterFragment`로 교체
- `components/PhysicsScene.tsx`: 요소별 `<div>` 생성 로직을 Voronoi 셀 기반으로 변경

이 변경으로 코드베이스가 크게 단순해지며, DOM 파싱 관련 버그(CLAUDE.md에 기록된 overlap/duplication, overflow clipping 문제)가 근본적으로 해결된다.
